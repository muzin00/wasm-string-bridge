# AI Agent 時代のセキュリティ と Wasm の必然性

強力な AI Agent の登場により、これまで暗黙に「人間の慎重さ」に依存していた
セキュリティモデルの限界が露呈し始めている。
Wasm のサンドボックス特性は、この時代の防御戦略の本命候補になりつつある。

## 構造: なぜ "今まで" のセキュリティが破綻するのか

```
強力な AI Agent の登場
   ↓
コードを実行する主体が "信頼できる人間" から "確率的に動く AI" に
   ↓
人間ベースのセキュリティモデル (権限管理・コードレビュー) では追いつかない
   ↓
既存システムの "想定外の経路" で脆弱性が露出 (= 負債の表面化)
   ↓
個別パッチで塞ぐのは限界 (攻撃面が広すぎる、AI が新しい経路を見つける)
   ↓
"封じ込めによる防御 (Containment)" への戦略転換
   ↓
強サンドボックス = Wasm が候補に浮上
```

## 表面化している "セキュリティ負債" の例

### 1. 過剰権限の問題

従来は「人間が慎重に使う」前提で与えていた権限が、AI Agent に渡るとリスクになる。
これは AI の問題というより、**そもそも "人間の慎重さ" に依存していた設計の脆さ** が露呈した構図。

### 2. プロンプトインジェクション

```
ユーザの質問 + 信頼できない外部データ (Web, ファイル等)
   ↓ どちらも同じ "テキスト" として AI に入る
AI が外部データの中の悪意ある指示に従ってしまう
   ↓
AI の権限で破壊的操作が実行される
```

伝統的な「**コードとデータの分離 (Code-Data Separation)** 」原則が、
自然言語ベースの AI では崩れている。

### 3. サプライチェーン攻撃の自動化

```
AI が "便利そう" と判断したパッケージを自動で導入
   ↓
そのパッケージが裏で credential を盗む
   ↓
従来は「人間がレビュー」していた防壁が消える
```

### 4. AI 生成コードの実行リスク

```
AI: "このスクリプトを実行すれば解決します"
ユーザ: (内容を読まずに) 実行
   ↓ AI が間違っていた、または攻撃を受けていた場合
ホスト OS で破壊的操作が走る
```

## 戦略の転換: Prevention → Containment

セキュリティ業界は伝統的に 2 つのアプローチを行き来してきた:

| 戦略 | 特徴 | 例 |
| --- | --- | --- |
| **Prevention** (予防 / 塞ぐ) | 脆弱性をなくす、入力を検証、権限を絞る | OWASP, secure coding, SAST |
| **Containment** (封じ込め) | 攻撃前提、被害範囲を制限 | サンドボックス, VM, 最小特権 |

AI 時代は **Containment への明確なシフト** が起きている:

1. **Prevention は完璧を要求する** — 1 つでも穴があれば破られる
2. **AI は無限に "創造的な穴" を見つける** — 人間が想定しない経路を発見
3. **Containment は失敗前提** — 攻撃成功しても被害範囲が限定される

「全部塞ぐのは現実的でない」という観測は、業界の合意に向かいつつある。

## なぜ Wasm がこの文脈で強いか

サンドボックス候補の比較:

| 技術 | 隔離強度 | 起動コスト | 言語サポート | AI Agent 用途 |
| --- | --- | --- | --- | --- |
| プロセス + seccomp | 中 | ms | 任意 | やや重い |
| chroot / unshare | 弱 | 低 | 任意 | 隔離不十分 |
| Docker | 中 | 100ms-数秒 | 任意 | 起動コスト過大 |
| microVM (Firecracker) | 強 | 数十-100ms | 任意 | クラウド FaaS で実用 |
| **Wasm Component** | **強** | **μs-ms** | **多言語** | **理論的に最適** |

Wasm の特異性:

### 1. Capability-based Security をデフォルトで持つ
何もしなければ何もできない (no I/O, no network)。
ホストが明示的に渡したものだけ使える。
→ AI が暴走しても、そもそもファイル/ネットワークにアクセスできない保証。

### 2. 起動が μs-ms オーダー
リクエストごとに新しい Sandbox を作って捨てるが現実的に可能。
→ "1 タスク = 1 Sandbox" のパターンが実装可能。

### 3. Shared-nothing
複数の AI 由来コードを同時に走らせても、互いに干渉できない。
→ マルチテナント AI 実行環境。

### 4. 言語非依存
Python の AI 生成コードも、Bash コマンドも、JS スニペットも、同じ Sandbox で扱える。
→ "AI が出力した何でも実行できる箱" が作れる。

### 5. 形式検証可能
Wasm のメモリモデルは形式検証可能で、サンドボックス自体の正しさが数学的に保証される。

| プロダクト | やっていること |
| --- | --- |
| Cloudflare Workers | Wasm でユーザコード隔離。AI 実行基盤としても採用拡大 |
| Fastly Compute@Edge | 同上 |

| WasmCloud | Wasm Component の分散実行基盤 |

特に **「LLM が生成したコードを安全に実行する」需要** が爆発的に伸びていて、
Wasm はその有力候補という構図が現実化している。

## Wasm が銀の弾丸ではない理由

公平のために制約も並べておく:

| 課題 | 内容 |
| --- | --- |
| エコシステムの未成熟 | 既存 Python/JS ライブラリの Wasm 対応が部分的 |
| デバッグ・観測性 | Sandbox 内部の可視化がまだ発展途上 |
| 性能 | 動的言語の Wasm はネイティブより遅い |
| 副作用の管理 | "WASI で何を許すか" の設計判断は依然として人間 |
| サイドチャネル | Spectre 等は別途対策必要 |
| 仕様の進行中 | WASI Preview 3 の async/HTTP/sockets が策定中 |

→ "現時点で最も筋の良い候補" だが、まだ成熟途上。今後 2-3 年で大きく変わる領域。

## 着地予測

「**Wasm が唯一解になる**」のではなく、用途で使い分けされる:

| 用途 | 技術 |
| --- | --- |
| 重い計算・既存資産 | コンテナ (Docker) |
| 完全隔離・既存 OS が必要 | microVM (Firecracker) |
| **軽量・短命・多言語・カスタムサンドボックス** | **Wasm** |

AI 時代の Sandbox の "新カテゴリ" を Wasm が獲得する、というのが現実的な予測。

## エンジニアとしての含意

この観察が正しいとすると:

- **今後 2-3 年で需要が大きく伸びる領域**
- **学習者がまだ少ない** = スキルの希少価値が高い
- 各層 (ランタイム / Component / WIT / WASI) が綺麗に設計されており、
  深い理解が長く役立つ
- AI セキュリティ分野とのクロスオーバーは hot な研究・実務領域

## 哲学的シフト

セキュリティ哲学の歴史的転換:

> **「完璧な城壁ではなく、被害を限定する区画」**

中世の城壁戦争 → 近代の縦深防御 と同じ進化が、AI の登場を契機にソフトウェアでも起きている。
Wasm はその主役候補の一つで、技術的特性がこの哲学に最適化されているように見える。

## 一言まとめ

> **Prevention の限界 → Containment への戦略転換 → Wasm のサンドボックス特性が AI 時代に最適化されている。**
> "注目されるかも" ではなく、**すでに注目され始めている** のが正確。
